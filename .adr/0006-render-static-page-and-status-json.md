# Render a Static Page and Status JSON

## Status

Accepted

## Context

The status page should be generated by GitHub Actions when issue state changes. It should not dynamically call GitHub from the browser, but the generated output should still be easy to test and inspect.

## Decision

The first working version will generate static assets: `index.html`, `status.json`, and CSS. `index.html` is the public status page. `status.json` is the derived status model used for verification and future integrations. CSS provides a clean readable presentation.

## Why

This gives users a visible status page while keeping the data model explicit. A JSON artifact makes tests and debugging easier without requiring a framework or a multi-page static site. Avoiding browser-side GitHub calls keeps GitHub Actions as the only place where GitHub issue data is fetched.

## Consequences

The renderer should separate status derivation from HTML rendering so tests can assert on `status.json` before visual output. Multi-page incident or system detail views can be added later if the single page becomes too dense.
